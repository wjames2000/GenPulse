### 2. 数据库环境说明

#### 2.1 数据库选型
- **数据库系统**：SQLite 3
- **版本要求**：≥ 3.35.0（支持 `STRICT` 表、`RETURNING` 子句）
- **存储位置**：`~/.ai-dev-pipeline/data/pipeline.db`
- **访问方式**：Go 语言通过 `github.com/mattn/go-sqlite3` 驱动访问

#### 2.2 选型理由
- **嵌入式**：无需独立服务，与桌面应用完美集成。
- **轻量高效**：资源占用低，启动快，适合单机应用。
- **全文检索**：通过 FTS5 扩展支持高效的中英文全文搜索。
- **JSON 支持**：原生 JSON 函数便于存储半结构化数据。
- **事务支持**：ACID 事务保证数据一致性。

#### 2.3 连接配置
```go
// 连接字符串示例
"file:~/.ai-dev-pipeline/data/pipeline.db?_journal_mode=WAL&_busy_timeout=5000&_foreign_keys=on"
```
- `_journal_mode=WAL`：启用 Write-Ahead Logging，提升并发性能。
- `_busy_timeout=5000`：锁等待超时 5 秒。
- `_foreign_keys=on`：强制启用外键约束。

#### 4.2 核心关系说明

- **项目配置**与**执行轨迹**：一对多关系，一个项目可执行多次流水线。
- **执行轨迹**与**Agent活动**：一对多关系，一次执行包含多个 Agent 的阶段活动。
- **技能索引**与**技能使用日志**：一对多关系，记录技能的每次使用情况。
- **情节记忆**为独立实体，通过 FTS5 实现全文检索，无外键关联。
- **自进化事件**记录技能生成、记忆更新等里程碑事件，关联执行 ID。

##### 5.1.7 episodic_memories（情节记忆表 - FTS5）

采用 SQLite FTS5 虚拟表实现全文检索。

```sql
CREATE VIRTUAL TABLE episodic_memories USING fts5(
    id UNINDEXED,
    task_type,
    description,
    steps,
    outcome,
    agent_role,
    timestamp,
    embedding,
    tokenize='porter unicode61'
);
```

| 字段名        | 类型 | 说明                                         |
| ------------- | ---- | -------------------------------------------- |
| `id`          | TEXT | 记忆唯一标识（UUID），不参与索引             |
| `task_type`   | TEXT | 任务类型：代码生成/调试/架构设计等           |
| `description` | TEXT | 任务描述（用于全文检索的主要内容）           |
| `steps`       | TEXT | 执行步骤摘要                                 |
| `outcome`     | TEXT | 结果与经验总结                               |
| `agent_role`  | TEXT | 记录该记忆的 Agent 角色                      |
| `timestamp`   | TEXT | 发生时间（ISO8601）                          |
| `embedding`   | TEXT | 向量嵌入的 Base64 编码（可选，用于语义检索） |

**FTS5 配置说明**：
- `tokenize='porter unicode61'`：支持英文词干提取和 Unicode 分词，兼顾中英文检索。

**影子表**：FTS5 自动创建 `episodic_memories_data`、`episodic_memories_idx` 等内部表。

### 7. 视图设计

#### 7.1 v_execution_summary（执行摘要视图）

提供执行轨迹的概览信息，便于仪表盘展示。

```sql
CREATE VIEW v_execution_summary AS
SELECT 
    et.id,
    et.project_name,
    et.start_time,
    et.end_time,
    et.status,
    et.total_tokens,
    et.total_cost,
    COUNT(DISTINCT aa.id) as agent_count,
    SUM(aa.tool_calls_count) as total_tool_calls
FROM execution_traces et
LEFT JOIN agent_activities aa ON et.id = aa.execution_id
GROUP BY et.id;
```

#### 7.2 v_skill_effectiveness（技能效力视图）

评估技能的使用效果。

```sql
CREATE VIEW v_skill_effectiveness AS
SELECT 
    si.name,
    si.version,
    si.usage_count,
    si.avg_success_rate,
    AVG(CASE WHEN sul.success = 1 THEN 1.0 ELSE 0.0 END) as recent_success_rate,
    MAX(sul.used_at) as last_used
FROM skills_index si
LEFT JOIN skill_usage_log sul ON si.name = sul.skill_name
GROUP BY si.name;
```

#### 7.3 v_agent_performance（Agent 性能视图）

统计各 Agent 的效率和成本。

```sql
CREATE VIEW v_agent_performance AS
SELECT 
    aa.agent_name,
    COUNT(*) as total_executions,
    AVG(aa.tokens_used) as avg_tokens,
    AVG(CAST((julianday(aa.end_time) - julianday(aa.start_time)) * 86400 AS REAL)) as avg_duration_sec,
    SUM(CASE WHEN aa.status = 'completed' THEN 1 ELSE 0 END) as success_count
FROM agent_activities aa
GROUP BY aa.agent_name;
```


### 8. 触发器设计

#### 8.1 自动更新 updated_at 字段

为需要记录更新时间的表创建触发器。

**示例：skills_index 表**
```sql
CREATE TRIGGER trg_skills_index_update 
AFTER UPDATE ON skills_index
BEGIN
    UPDATE skills_index SET updated_at = CURRENT_TIMESTAMP WHERE name = NEW.name;
END;
```

**示例：project_configs 表**
```sql
CREATE TRIGGER trg_project_configs_update 
AFTER UPDATE ON project_configs
BEGIN
    UPDATE project_configs SET updated_at = CURRENT_TIMESTAMP WHERE name = NEW.name;
END;
```

#### 8.2 级联更新技能使用计数

当 `skill_usage_log` 插入新记录时，自动更新 `skills_index` 的 `usage_count` 和 `last_used_at`。

```sql
CREATE TRIGGER trg_skill_usage_log_insert
AFTER INSERT ON skill_usage_log
BEGIN
    UPDATE skills_index 
    SET usage_count = usage_count + 1,
        last_used_at = NEW.used_at
    WHERE name = NEW.skill_name;
END;
```

#### 8.3 自动清理（可选）

定期清理超过 90 天的旧执行轨迹，可通过应用逻辑实现，不强制使用触发器。

### 10. 安全与备份策略

#### 10.1 数据安全
- **敏感信息**：API Key 不存储在数据库中，使用操作系统 Keyring 管理。
- **文件权限**：数据库文件权限设置为 `0600`（仅当前用户可读写）。
- **SQL 注入防护**：所有数据库操作使用参数化查询（Go `database/sql` 占位符）。

#### 10.2 备份与恢复
- **自动备份**：每次应用启动时，检查并创建数据库备份（`pipeline_backup_{date}.db`），保留最近 7 个备份。
- **手动导出**：前端提供“导出数据”功能，将执行历史、技能等导出为 JSON 文件。
- **恢复机制**：支持从备份文件恢复，需用户确认。

#### 10.3 数据清理
- 提供设置项：自动清理超过 N 天的执行轨迹（默认 90 天）。
- 清理时同步删除关联的 `agent_activities` 和 `tool_call_logs`（通过外键级联）。

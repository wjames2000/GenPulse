# 项目需求说明书

## AI 软件项目开发流水线（桌面版）


### 一、项目概述

#### 1.1 项目名称

**AI 软件项目开发流水线（桌面版）** — 基于 Google Genkit 的多 Agent 协作式全自动软件开发平台

#### 1.2 项目背景

当前 AI 辅助软件开发工具（如 GitHub Copilot、Cursor、Claude Code 等）主要以“助手”形态存在，提供代码补全、问答、单次生成等能力。但实现“从 0 到 1 全自动写项目”仍然面临以下挑战：

- **缺乏流水线编排**：现有工具无法将需求分析、架构设计、编码实现、测试、部署等环节自动化串联。
- **Agent 协作能力弱**：不同角色（架构师、前端、后端、测试等）之间的协作需要人工协调。
- **缺乏可观测性**：难以监控多 Agent 并行执行时的进度、资源消耗和决策过程。
- **本地文件操作受限**：大多数 AI 编码工具通过 IDE 扩展运行，对文件系统和 Git 的直接操作能力有限。

#### 1.3 项目目标

开发一款**桌面版 AI 软件项目开发流水线**软件，具备以下核心能力：

1. **全自动项目生成**：用户输入需求描述，系统自动完成从项目初始化到可运行代码的全流程生成。
2. **多 Agent 协作**：支持配置多个角色 Agent（产品经理、架构师、前端、后端、测试等），各司其职、协同工作。
3. **Agent 能力扩展**：通过 MCP 协议和 Skills 机制扩展 Agent 的工具调用能力。
4. **实时监控**：可视化监控每个 Agent 的执行状态、推理过程和资源消耗。
5. **本地操作能力**：直接操作文件系统（生成、读取、复制、移动、删除）和 Git 仓库。

#### 1.4 首个核心场景

**“全自动从 0 到 1 写项目”** ：用户输入项目需求描述 → 系统自动完成需求分析 → 技术方案设计 → 项目脚手架生成 → 代码实现 → 测试编写 → 项目运行验证，全程无需人工干预。

#### 2.3 为什么选择 Genkit

Genkit 是 Google 开源的生产级 AI 框架，具备以下优势：

- **统一 API**：支持 Google AI（Gemini）、Vertex AI、OpenAI、Anthropic、Ollama 等多家模型提供商。
- **类型安全的 Flow**：可定义带类型检查的 AI 工作流，确保输入输出可预测、可测试。
- **原生多 Agent 支持**：通过将 Agent 作为 Tool 实现层级化 Agent 编排。
- **丰富的开发工具**：CLI + Developer UI 支持调试、追踪、监控。
- **MCP 深度集成**：原生支持 Model Context Protocol，可扩展任意工具能力。
- **Node.js 与 Go 双语言支持**：本项目选用 Node.js 版，与 Electron 生态无缝集成。

**与主流框架对比**：

| 对比维度          | Genkit                | LangChain        | CrewAI               | AutoGen      |
| ----------------- | --------------------- | ---------------- | -------------------- | ------------ |
| **多 Agent 架构** | 层级化 Agent-as-Tool  | LangGraph 状态机 | 角色化 Agent 团队    | 对话式 Agent |
| **生产可用性**    | 高（Google 内部验证） | 中等             | 中高（token 消耗大） | 中等         |
| **类型安全**      | 原生支持              | 部分支持         | 弱                   | 弱           |
| **MCP 支持**      | 深度集成              | 插件支持         | 有限                 | 有限         |
| **开发工具**      | CLI + UI 齐全         | 分散             | 有限                 | 有限         |

Genkit 的层级化多 Agent 架构通过将每个专业 Agent 封装为工具（Tool），由编排 Agent 统一调度，实现了更清晰的分层和更低的 token 消耗。

#### 3.2 Agent 管理与编排模块

**功能描述**：核心的 Agent 定义、编排和执行引擎。

**3.2.1 Agent 角色定义**

系统内置以下 Agent 角色（可扩展）：

| Agent 角色                        | 职责                                     | 工具集                       |
| --------------------------------- | ---------------------------------------- | ---------------------------- |
| **Orchestrator**（编排 Agent）    | 接收用户需求，分解任务，调度各专业 Agent | 调用其他 Agent，管理执行流程 |
| **Product Manager**（产品经理）   | 分析需求，输出 PRD 文档、用户故事        | 文件写入、需求模板           |
| **Architect**（架构师）           | 设计技术架构、模块划分、数据模型         | 文件写入、架构模板           |
| **Frontend Dev**（前端开发）      | 生成前端代码（组件、页面、样式）         | 文件读写、npm 操作、代码模板 |
| **Backend Dev**（后端开发）       | 生成后端代码（API、服务、数据层）        | 文件读写、数据库 Schema 生成 |
| **Full-stack Dev**（全栈开发）    | 处理小型项目的全栈实现                   | 全栈工具集                   |
| **QA Engineer**（测试工程师）     | 编写单元测试、集成测试                   | 测试框架调用、测试运行       |
| **DevOps Engineer**（运维工程师） | 项目初始化、依赖安装、启动脚本           | Git、npm、命令行             |
| **Reviewer**（代码审查）          | 审查生成代码的质量、安全性               | 代码分析工具                 |

**3.2.2 Agent 编排机制**

基于 Genkit 的层级化多 Agent 架构：

- 每个专业 Agent 通过 `ai.definePrompt()` 定义，包含专属的系统提示词、模型配置和可用工具。
- 通过 `asTool()` 方法将 Agent 转化为可被其他 Agent 调用的工具。
- Orchestrator Agent 作为主入口，根据任务需求调用对应的专业 Agent。
- 支持 `ai.chat()` 会话机制，保持跨 Agent 调用的上下文连续性。

#### 3.3 Skills 机制

**功能描述**：通过 Skills 扩展 Agent 的专业能力和领域知识。

**Skill 定义**：

每个 Skill 包含：
- **名称与描述**：供 Orchestrator 识别何时调用
- **触发条件**：自然语言描述的适用场景
- **专业知识库**：领域特定的最佳实践、代码模板、架构模式
- **工具扩展**：Skill 专属的工具函数
- **示例 Few-shot**：提升模型执行质量的示例

**内置 Skills**：

| Skill 名称                 | 功能                         |
| -------------------------- | ---------------------------- |
| `react-component-builder`  | 生成符合规范的 React 组件    |
| `api-endpoint-generator`   | 生成 RESTful API 端点代码    |
| `database-schema-designer` | 设计数据库 Schema 和迁移脚本 |
| `unit-test-writer`         | 编写 Jest/Vitest 单元测试    |
| `git-workflow`             | 执行 Git 初始化和提交操作    |
| `dependency-manager`       | 分析并安装项目依赖           |
| `code-review-security`     | 安全漏洞扫描和修复建议       |

#### 3.4 MCP 集成模块

**功能描述**：通过 Model Context Protocol 接入外部工具和服务，无限扩展 Agent 能力。

**MCP 功能点**：

| 编号  | 功能           | 说明                                                         |
| ----- | -------------- | ------------------------------------------------------------ |
| F-4.1 | MCP Client     | 作为客户端连接外部 MCP Server，使用其提供的工具、资源和提示词 |
| F-4.2 | MCP Server     | 将本系统的工具暴露为 MCP Server，供其他 AI 工具调用          |
| F-4.3 | 多 Server 管理 | 通过 MCPHost 管理多个 MCP Server 连接，支持动态启停          |
| F-4.4 | 工具发现       | 自动发现并注册 MCP Server 提供的工具                         |
| F-4.5 | 命名空间隔离   | 为不同 MCP Server 的工具创建命名空间，避免冲突               |

**推荐接入的 MCP Server**：

- **GitHub MCP**：操作 Issues、PR、仓库管理
- **Filesystem MCP**：增强文件系统操作（已通过 genkitx-fs 内置支持）
- **Puppeteer MCP**：浏览器自动化、E2E 测试
- **Postgres / SQLite MCP**：数据库操作
- **Figma MCP**：设计稿转代码

### 四、模型选型建议

#### 4.1 初期核心模型：以 Gemini 为主导

**推荐模型配置**：

| Agent 角色      | 推荐模型             | 理由                                               |
| --------------- | -------------------- | -------------------------------------------------- |
| Orchestrator    | **Gemini 2.5 Flash** | 低延迟、低成本，适合高频调度任务                   |
| Product Manager | **Gemini 2.5 Pro**   | 强推理能力，适合复杂需求分析                       |
| Architect       | **Gemini 2.5 Pro**   | 顶尖的编码和架构设计能力                           |
| Frontend Dev    | **Gemini 2.5 Pro**   | WebDev Arena 排名第1（1420 ELO），前端生成能力最强 |
| Backend Dev     | **Gemini 2.5 Pro**   | SWE-Bench Verified 63.8%，代码生成能力强           |
| QA Engineer     | **Gemini 2.5 Flash** | 测试生成任务相对标准化，Flash 足够胜任             |
| Reviewer        | **Gemini 2.5 Flash** | 代码审查可并行执行，Flash 更经济                   |
| DevOps Engineer | **Gemini 2.5 Flash** | 脚本化任务，Flash 足够胜任                         |

**Gemini 作为核心模型的优势**：

1. **顶尖编码能力**：Gemini 2.5 Pro 在多个编码基准测试中排名第一，WebDev Arena 1420 ELO 超越 Claude 3.7 Sonnet 的 1357；在 LiveCodeBench 推理任务中，Gemini 2.5 Flash 得分 51%，超越 Claude 3.7 Sonnet Thinking（47%）和 GPT-4.1（46%）。
2. **超大上下文窗口**：2M tokens 原生上下文，可一次性处理整个项目代码库。
3. **多模态能力**：支持图文理解、视频转代码，可基于设计稿生成代码。
4. **Genkit 原生集成**：与 Genkit 框架深度整合，开发体验最流畅。
5. **免费/低价入门**：Gemini 2.5 Flash 有慷慨的免费额度，适合初期验证。

#### 4.2 多模型策略

**降级与备用机制**：

- 当 Gemini API 不可用时，自动切换到 Anthropic Claude 3.7 Sonnet 或 OpenAI GPT-4o。
- 利用 Genkit 的统一 API，通过配置即可切换模型提供商。

**成本优化策略**：

- 使用 Gemini 2.5 Flash 处理大量简单任务（如代码审查、测试生成）。
- 仅在关键决策节点（架构设计、复杂逻辑实现）使用 Gemini 2.5 Pro。
- 本地开发验证阶段可接入 Ollama 运行开源模型（如 Qwen-Coder），零成本调试。

### 八、安全与隐私设计

#### 8.1 API Key 管理

- 使用操作系统原生安全存储：
  - macOS: Keychain Services
  - Windows: Credential Manager
  - Linux: libsecret / kwallet
- 所有 API Key 仅在内存中解密使用，不写入明文配置文件。

#### 8.2 工作区隔离

- 每个项目绑定唯一的根目录路径。
- Agent 通过 `fs_*` 工具操作时，自动将路径限定在项目根目录内。
- 提供“安全模式”，禁止访问 `.env`、`.git/credentials` 等敏感文件。

#### 8.3 命令执行沙箱

- 维护命令白名单，仅允许执行预定义的命令类别。
- 危险命令（如 `rm -rf`、`format`）需用户二次确认。
- 记录所有命令执行的完整参数和输出。

#### 8.4 隐私承诺

- 所有代码生成和数据存储均在本地完成。
- 仅 API 请求发送至模型服务商（Gemini API 等）。
- 不采集任何遥测数据，除非用户主动开启。

### 十、风险与应对

| 风险                | 影响         | 概率 | 应对措施                                                   |

| 模型输出不稳定      | 代码质量波动 | 中   | 使用 Genkit 的类型安全输出 + JSON Schema 约束 + 自动重试   |
